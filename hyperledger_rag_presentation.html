O que é RAG?

RAG (Retrieval-Augmented Generation) é uma técnica que combina recuperação de informações com geração de texto para fornecer respostas mais precisas e contextualizadas.

Fases do RAG

0. Fase de Coleta de Dados (Setup)

Documentos são coletados de diversas fontes (websites, PDFs, bases de dados, etc.). Neste projeto, o crawler coleta automaticamente páginas da documentação do Hyperledger Fabric através de web scraping.

1. Fase de Indexação (Offline)

Documentos são divididos em chunks pequenos, convertidos para embeddings vetoriais e armazenados em um banco de dados vetorial para busca rápida.

2. Fase de Recuperação (Retrieval)

Quando o usuário faz uma pergunta, ela é convertida em embedding e comparada com os embeddings dos documentos para encontrar os trechos mais relevantes.

3. Fase de Geração (Generation)

Os documentos recuperados são passados como contexto para um modelo de linguagem, que gera uma resposta baseada no contexto fornecido.

O que é Hyperledger?

Hyperledger é um projeto de código aberto mantido pela Linux Foundation que fornece ferramentas e bibliotecas para desenvolvimento de aplicações blockchain empresariais.

Diferenças: Scraper e Crawler

Scraper

Objetivo: Extrair dados de uma página HTML específica

Uso: Coleta de dados pontuais, parsing de conteúdo

Exemplo: BeautifulSoup, Selenium em uma URL

Crawler

Objetivo: Navegar automaticamente entre páginas seguindo links

Uso: Indexação de sites, coleta massiva de dados

Exemplo: Google Bot, projeto Hyperledger RAG

Resumo: Scraper extrai dados de uma página → Crawler navega automaticamente entre páginas seguindo links

Motor Crawler do Projeto

O crawler foi implementado usando BeautifulSoup para parsing HTML e BFS (Breadth-First Search) para navegação entre páginas.

Motor RAG do Projeto

O RAG foi implementado integrando FAISS para recuperação, TinyLlama como LLM e LangChain para orquestração.

Fluxo de Geração

0. Coleta de Dados

Documentos são coletados de fontes externas (crawler coleta páginas HTML da documentação)

1. Embedding da Query

A pergunta é convertida em um vetor usando o mesmo modelo de embeddings (sentence-transformers)

2. Busca Vetorial

FAISS busca os k documentos mais similares usando similarity_search

3. Context Window

Os documentos recuperados são inseridos como contexto no prompt do LLM

4. Geração de Resposta

TinyLlama gera uma resposta baseada no contexto fornecido

Classe RAGChain

class RAGChain: def __init__(self, llm, vector_store): # Integra LLM com retriever FAISS self.qa_chain = RetrievalQA.from_chain_type( llm=llm.llm, chain_type="stuff", retriever=vector_store.db.as_retriever( search_kwargs={"k": 2} ) ) def ask(self, query): response = self.qa_chain.invoke(query) return response['result']
